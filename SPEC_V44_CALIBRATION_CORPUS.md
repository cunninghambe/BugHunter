# SPEC — v0.44 "Public calibration corpus + `bughunter calibrate`"

**Status:** Draft 1 — ready for `@coder` assignment · **Author:** `@architect` (Opus, ultrathink) · **Date:** 2026-04-30 · **Predecessor:** `SPEC_PATH_TO_EXHAUSTIVE.md` §6.3 (public calibration benchmark) and §9 Phase F. · **Depends on:** v0.27 `bugIdentity` (cluster-to-gold matching key — `BugCluster.bugIdentity` is the only stable identifier that survives across runs and across machines), v0.26 `DETECTOR_REGISTRY` (per-kind precision/recall reporting iterates the registry, not an ad-hoc kind list, so deferred and wired kinds are handled differently and a new kind cannot silently disappear from the report). · **Sibling:** v0.33 `bughunter self-test` — the *internal* fixture-based smoke (one-bug-per-kind diagnostic, lives inside this repo at `fixtures/bughunter-self-deliberate-bugs/`); v0.44 is the **public** benchmark (real-shaped vibe-coded apps, lives in a separate repo `cunninghambe/BugHunter-bench`, drives README numbers).

This spec implements the **trust** axis from the path-to-exhaustive roadmap. Today, BugHunter's "comprehensive" claim is asserted, not demonstrated — there are no published precision/recall numbers, no curated corpus, and no reproducible way for a third party to verify the detector palette. V44 ships (a) a separate public repo `BugHunter-bench` containing 5 deliberately-buggy real-shaped apps with hand-curated `gold-standard.jsonl` per app, (b) a new `bughunter calibrate` subcommand that compares emitted clusters to gold via `bugIdentity` matching, computes per-kind precision/recall/F1 and emits `calibration-report.json`, and (c) a CI workflow that publishes the latest numbers into BugHunter's main `README.md` after every PR merge.

The implementation is bounded: ~700 lines of code in BugHunter (one new CLI command, one new module, one CI script), ~5,000 lines of fixture code in the bench repo (~1,000 per app), ~500 lines of gold-standard JSONL across all five fixtures. The hard work is at the seam where emitted clusters and gold entries are matched: V27's `bugIdentity` is the canonical key, but V44 adds a fallback "structural-equivalent" match so a fixture maintainer can author gold *before* running BugHunter and compute identities (gold authors do not need a working BugHunter to write gold).

---

## 1. Objective

Add a public, reproducible calibration benchmark so a third party can verify BugHunter's per-kind precision and recall.

| Question | Answered by |
|---|---|
| What is BugHunter's true positive rate on each BugKind? | `bughunter calibrate --app <app> --gold gold-standard.jsonl` → per-kind precision/recall/F1 in `calibration-report.json` |
| Did this PR regress detection on any kind? | CI runs `calibrate` against all five bench apps, posts a delta on the PR |
| Are deferred kinds correctly silent (no false positives)? | Calibrator iterates the V26 `DETECTOR_REGISTRY`; deferred kinds with `actual: 0` and `gold: 0` are reported as `status: 'silent_as_expected'` |
| Where does the 0.85 / 0.80 acceptance bar sit per kind? | `acceptance-thresholds.json` in this repo, consumed by `--enforce-thresholds` |
| What numbers does the README publish? | `README.md` `<!-- BEGIN CALIBRATION -->...` block, regenerated by CI from the latest `calibration-report.json` |

**In scope (V44):**

- New companion repo `cunninghambe/BugHunter-bench` (Part 1) — 5 fixture apps, one `gold-standard.jsonl` per app, plus a top-level `MANIFEST.json` enumerating apps and the BugKind coverage matrix.
- New CLI command `bughunter calibrate` (Part 2) — runs BugHunter against a bench app, matches emitted clusters to gold, computes precision/recall/F1 per BugKind, emits `calibration-report.json`.
- New `acceptance-thresholds.json` — per-kind precision/recall thresholds, with a default of `precision ≥ 0.85, recall ≥ 0.80` and per-kind overrides for inherently noisier kinds (`visual_anomaly`, `axe_color_contrast_strong`).
- New CI workflow `.github/workflows/calibrate.yml` in the BugHunter main repo — on every PR-to-main and merged-to-main, clones bench, runs calibrate against all five apps, posts the delta as a PR comment, and on merge regenerates the `README.md` calibration block.
- Acceptance test on the bench corpus: per-kind precision ≥ 0.85, recall ≥ 0.80 across the union of all five apps for every wired BugKind that has at least 3 gold entries.

**Out of scope (deferred):**

- Growing the corpus beyond 5 apps. Industry SAST benchmarks use 50–200; 5 is a starting point. Spec §11 records this and commits to growing to 20+ over Phase F follow-ups.
- Kind-by-kind severity thresholds. V44's acceptance bar is uniform precision ≥ 0.85 / recall ≥ 0.80 with per-kind overrides; severity-weighted F-beta is deferred.
- Auto-fix calibration. V44 measures detection only; whether `bughunter fix-coord` actually fixes the gold-known bugs is a separate spec (V45 or later).
- Cross-version regression history. V44 publishes the latest numbers; storing a multi-run history of `calibration-report.json` and graphing trends is deferred.
- Adversarial fixtures (deliberately bug-free apps that should produce zero clusters). Useful for false-positive measurement at scale; defer to V45 once the basic corpus is stable.
- Public-leaderboard-against-other-tools (SonarCloud, DeepSource, Codacy on the same corpus). Out of scope for V44; possible future content-marketing project.
- Translation of fixture UIs into non-English locales. Bench apps are English-only; i18n calibration is in §3.2 of the path-to-exhaustive roadmap and will live in a sibling spec.
- Proprietary fixtures derived from real customer apps. The bench corpus is 100% synthetic to avoid licensing entanglement; "apps that look like real ones" not "real apps."

**Acceptance target:**

After landing V44 on `main` and merging the bench repo:

1. `bughunter calibrate --app vibe-todo --gold ./gold-standard.jsonl --enforce-thresholds` exits 0 for each of the five apps.
2. Aggregating across all five apps, every wired BugKind with ≥ 3 gold entries has precision ≥ 0.85 and recall ≥ 0.80.
3. `README.md` contains a `<!-- BEGIN CALIBRATION -->...<!-- END CALIBRATION -->` block populated with the latest per-kind numbers, last-updated timestamp, and bench-repo commit SHA.
4. The CI workflow runs on PR-to-main and the action takes ≤ 25 minutes wall-clock total (5 apps × ≤ 5 min each).
5. A maintainer can add a new gold entry by editing one JSONL line and re-running `calibrate` locally without recompilation.

### 2.2 Patterns to follow

- **Discriminated unions for match outcomes.** Each gold entry produces a typed `MatchOutcome`: `{ kind: 'true_positive', goldId, clusterId, matchVia: 'bugIdentity' | 'structural' }`, `{ kind: 'false_negative', goldId }`, `{ kind: 'false_positive', clusterId }`. Drives report formatting via exhaustive switch. Discriminated unions over flag-tuples — agents succeed near 100% with typed contracts.
- **Single source of truth for kind enumeration.** Iterate `DETECTOR_REGISTRY` (V26), not a hardcoded list of BugKinds. New kinds enter the report automatically; deferred kinds get `status: 'expected_silent'` not `precision: NaN`.
- **Two match strategies, layered.** Primary: exact `bugIdentity` match (sha256-derived). Fallback: **structural** match — `(kind, normalizedLocation, normalizedMessage)` tuple comparison, used only when a gold entry has no `bugIdentity` field (gold author hadn't run BugHunter at gold-write time). Structural match is documented as **second-class**: report flags it as `matchVia: 'structural'` and the bench-repo CI must promote structural matches to bugIdentity matches by re-recording on pass — see §6.3.
- **Per-kind reporting.** The report is `Record<BugKind, KindReport>` plus a global aggregate. Missing kinds (no gold entries, no clusters minted) show as `status: 'no_data'`, never as `precision: 0`.
- **Fixture isolation.** Each bench app boots on its own port (`vibe-todo` 4101, `vite-shop` 4102, `vue-board` 4103, `next-blog` 4104, `astro-saas` 4105). Calibrator never assumes a default; the app's `bughunter.config.json` declares the port. CI starts each app sequentially to avoid port collisions and resource contention.
- **No network dependency.** Bench apps are 100% offline (no third-party API calls, no CDN requirements). Reproducibility demands it.
- **Gold authoring ergonomics.** A gold entry is a single JSONL line. No comments inside JSONL (it is a strict JSON-lines format). Per-app `gold-standard.md` documents the human-readable rationale for each entry; CI verifies gold-standard.jsonl line count equals gold-standard.md `## Gold-N` heading count.

### 2.3 DO NOT

- Do **not** put bench apps inside the BugHunter main repo. They go in the separate `cunninghambe/BugHunter-bench` repo. Mixing them creates an honesty problem: a contributor can hand-tune the detector to pass the corpus, defeating the calibration value.
- Do **not** import bench fixtures via git submodule. CI clones the bench repo at a pinned SHA in the workflow YAML; locally, the user clones it themselves. Submodules add ceremony for casual contributors.
- Do **not** reuse `fixtures/` (`a11y-bad`, `seo-bad`, `pen-bad`, etc.) as bench apps. Those are internal smoke fixtures (V33). The bench corpus must be real-shaped vibe-coded apps with realistic stack mixes (Vite + React, Next.js, Vue, Astro), not minimal one-bug-per-page test cases.
- Do **not** auto-update gold from a passing run. Gold is hand-curated. A new bug found by the detector but not in gold must be human-reviewed and added (or marked as a false positive). Auto-promotion converts "calibration" into "tautology."
- Do **not** write a parallel cluster-output format. The calibrator reads the existing `summary.json.clusters` array and the existing `bugs.jsonl` JSONL stream. Match the V27 shape, not a new shape.
- Do **not** introduce a runtime dep for hashing or matching. `node:crypto` (already in use for `bugIdentity`) and native `Map`/`Set` are sufficient.
- Do **not** reach inside the run lifecycle from `calibrate.ts`. Invoke the public `runRun(...)` entrypoint and read its summary path; do not re-emit clusters from a half-built `RunState`.
- Do **not** silently treat deferred kinds as failures. A deferred kind with gold count of 0 is correct; with gold count > 0 is a gold-authoring error and the calibrator must surface it.
- Do **not** publish numbers from a partial run. If the underlying run is partial (e.g., crashed mid-cluster), `calibrate` exits non-zero with a clear "underlying run failed; not publishing numbers" message.
- Do **not** include real user data, real API keys, real domain content, or any third-party trademarks in bench apps. Synthetic, public-domain content only.

### 3.4 BugKind coverage rule

Every wired BugKind must have **at least 1** gold entry across the corpus, and at least one BugKind per category (security, perf, a11y, SEO, race, console-error, network) must have **3+** gold entries to support a meaningful per-kind precision/recall (3 is the minimum for a stable rate; below that, precision/recall numbers are reported but flagged `lowConfidence: true` in the report).

Every deferred BugKind must have **exactly 1** gold entry whose `expected: 'detector_silent'` field marks it as a "gap demonstrator." The calibrator asserts BugHunter does **not** mint a cluster matching it; if it does, that is a false positive against a deferred kind, which is a coverage regression.

## 4. Gold-standard format

### 4.1 `gold-standard.jsonl` line shape

One JSON object per line, no comments, no blank lines.

```jsonc
{
  "goldId": "vibe-todo-001",
  "kind": "race_condition_optimistic_revert",
  "expected": "detector_fires",
  "bugIdentity": "9f7a3e2b8c1d4505",
  "structuralMatch": {
    "kind": "race_condition_optimistic_revert",
    "normalizedLocation": "/dashboard:list-add",
    "normalizedMessage": "todo-add-then-revert",
    "suspectedFile": "src/components/TodoList.tsx"
  },
  "rationale": "Adds todo, optimistic UI flips, server returns 500, UI must revert but does not — leaves orphan todo in list.",
  "humanRepro": [
    "Login as owner",
    "On /dashboard, click 'Add todo' with title 'breaking-task'",
    "Server returns 500 (deterministic — title 'breaking-task' triggers it)",
    "Observe the todo remains in the list with no error toast"
  ],
  "minClusterSize": 1,
  "addedInBenchVersion": "0.1.0"
}
```

| Field | Required | Meaning |
|---|---|---|
| `goldId` | yes | Stable, human-readable id, format `<app-id>-<3-digit-counter>`. Renaming bumps benchVersion. |
| `kind` | yes | A BugKind from V26 registry. Validated against the registry at `calibrate` start. |
| `expected` | yes | Discriminated tag: `'detector_fires'` (positive case) or `'detector_silent'` (negative case for deferred kinds and adversarial scenarios). |
| `bugIdentity` | optional | The 16-char hex from V27. Authoritative when present. Filled in after the first passing calibrate run via `--record-identities`. |
| `structuralMatch` | required if `bugIdentity` is absent | Fallback match shape. Tuple of `(kind, normalizedLocation, normalizedMessage)` plus optional `suspectedFile` for diagnostic; never used for matching. |
| `rationale` | yes | One-sentence explanation. Reviewed by humans on PR. |
| `humanRepro` | yes | Steps to reproduce manually. Required for gold authoring; supports adversarial review. |
| `minClusterSize` | optional, default 1 | The minimum size of a matching cluster. Some bugs (XSS, SEO) typically produce 1 occurrence; others (n+1, console_error) cluster many. |
| `addedInBenchVersion` | yes | Tracks when this gold entry was added. Used to attribute regressions. |

### 4.2 `gold-standard.md` companion

Co-located narrative file. One `## Gold-N` section per JSONL line, in the same order. CI invariant: `wc -l gold-standard.jsonl == grep -c '^## Gold-' gold-standard.md`. The narrative file is the human-readable artifact for PR review; the JSONL file is machine-readable.

### 4.3 Authoring workflow

```
1. Author writes a deliberate bug in the app code, with a clear repro.
2. Author writes a gold-standard.jsonl line WITHOUT bugIdentity (it is unknown).
3. Author writes the gold-standard.md companion section.
4. Author runs `bughunter calibrate --app <app> --gold gold-standard.jsonl --record-identities`
   which structurally matches the gold to a cluster and rewrites gold-standard.jsonl
   with the bugIdentity filled in.
5. Author commits the rewritten gold-standard.jsonl. Future runs match via bugIdentity (faster, exact).
6. PR reviewer reads gold-standard.md and approves.
```

If structural matching is ambiguous (multiple clusters match), `--record-identities` exits non-zero and prints all candidates; the author disambiguates by tightening `structuralMatch.normalizedLocation` or `normalizedMessage`.

## 5. Calibration algorithm

### 5.1 Pipeline

```
Input: bench-app path, gold path, optional thresholds, optional --enforce-thresholds
Output: calibration-report.json + exit code

Step 1 — Validate gold
  Parse JSONL. Validate each line via Zod schema (3.4 above). Fail fast on invalid lines.
  Cross-check every kind against V26 DETECTOR_REGISTRY. Unknown kinds → fatal error.
  Cross-check every deferred-kind gold has expected: 'detector_silent'. Mismatch → fatal.

Step 2 — Boot the bench app
  Run app's calibrate.bootScript. Poll healthCheckUrl until 200 or healthCheckTimeoutMs.
  On timeout → fatal, with stderr from the boot script.

Step 3 — Run BugHunter
  Invoke runRun() against the bench app. Honor app's bughunter.config.json budget.
  Read the resulting summary.json from runDir(projectDir, runId).
  If summary.partial === true → fatal ("partial run cannot be calibrated").

Step 4 — Match clusters to gold
  Build two indexes:
    - byBugIdentity: Map<string, BugCluster>  (clusters keyed on cluster.bugIdentity)
    - byStructural: Map<string, BugCluster[]> (key: kind|normalizedLocation|normalizedMessage)
  For each gold entry:
    if gold.bugIdentity is set:
      look up byBugIdentity. If found → MatchOutcome { kind: 'true_positive', via: 'bugIdentity' }.
      If not found AND gold.expected === 'detector_fires' → MatchOutcome { kind: 'false_negative' }.
      If not found AND gold.expected === 'detector_silent' → MatchOutcome { kind: 'true_negative' }.
    else:
      look up byStructural with gold.structuralMatch. If exactly one cluster → match it.
      If many → fatal (gold ambiguity — author must disambiguate).
      If none AND gold.expected === 'detector_fires' → MatchOutcome { kind: 'false_negative' }.
      If none AND gold.expected === 'detector_silent' → MatchOutcome { kind: 'true_negative' }.
  Any cluster not consumed by a true_positive → MatchOutcome { kind: 'false_positive' }.

Step 5 — Aggregate per-kind
  For each kind in DETECTOR_REGISTRY:
    tp = count of true_positives with cluster.kind == kind
    fp = count of false_positives with cluster.kind == kind
    fn = count of false_negatives with gold.kind == kind
    tn = count of true_negatives with gold.kind == kind  (only meaningful for deferred kinds)
    precision = tp / (tp + fp), or 1.0 if tp+fp == 0 and (kind is deferred and tn > 0)
    recall    = tp / (tp + fn), or 1.0 if tp+fn == 0 and (kind is deferred)
    f1        = 2 * p * r / (p + r), or NaN if p+r == 0
    lowConfidence = (tp + fn) < 3   # too few gold entries for a stable rate

Step 6 — Threshold evaluation
  If --enforce-thresholds:
    For each kind: if precision < threshold.precision OR recall < threshold.recall AND not lowConfidence:
      kind reported as 'below_threshold'; exit code 1
    If kind is 'below_threshold' AND lowConfidence: warning, not failure (the gold is too sparse).

Step 7 — Emit calibration-report.json
  Write to <projectDir>/.bughunter/calibration/<benchVersion>-<isoDate>/calibration-report.json
  Print one-line human summary to stdout.
```

### 5.2 Edge cases

**EC-1. Same gold entry matched by both bugIdentity and structural.** Prefer bugIdentity always. The gold's `bugIdentity` is the canonical key; structural is fallback only when bugIdentity is absent.

**EC-2. Cluster has bugIdentity matching a gold from a different app.** Impossible by construction: `bugIdentity = sha256(projectName + ' ' + signatureKey)`, and projectName is per-app. If it occurs, it indicates a bug in V27 minting and is fatal.

**EC-3. Multiple clusters share the same bugIdentity in one run.** Should not happen — V27 mints one cluster per (projectName, signatureKey). If it occurs, fatal: there is a clustering bug.

**EC-4. Gold entry references a kind that exists in registry but is deferred, with expected: 'detector_fires'.** Fatal at step 1: deferred kinds can only have `expected: 'detector_silent'`. The author must either (a) wire the detector first, then re-author the gold, or (b) mark the gold as `detector_silent` to demonstrate the gap.

**EC-5. App boots but health check returns 200 before the app is actually ready (race in bootScript).** Author's responsibility. The healthCheckUrl must be a "real" readiness probe — for SPAs, a tag in the rendered HTML; for backends, a `/healthz` that touches the DB.

**EC-6. `runRun` succeeds but emits zero clusters.** Compatible with a bug-free run; the calibrator's report shows every gold's `expected: 'detector_fires'` as false_negative (recall: 0) and every `detector_silent` as true_negative. If acceptance thresholds are enforced and recall is below 0.80, calibrate exits 1.

**EC-7. Gold count < 3 for some kind, calibrate fails on that kind.** Reported as `belowThreshold` AND `lowConfidence: true`. Acceptance check: with `--enforce-thresholds`, low-confidence kinds are downgraded to warnings, not failures. CI prints them in a "needs more gold" section.

**EC-8. Bench app already running on its port (port collision with developer's local stack).** boot.sh must `lsof -i :<port>` and refuse to overwrite. The CI environment is clean; locally, the user is responsible. Non-fatal locally — message-and-skip with exit 2.

**EC-9. `summary.json.clusters[].bugIdentity` is undefined (V27 not yet wired in user's branch).** Fatal at step 4: V44 hard-requires V27. Error message points to the V27 spec and explains why.

**EC-10. Gold author adds a new entry; structural match resolves to a real cluster, but the cluster's bugIdentity differs from a previous run's cluster for the same gold.** Indicates a regression in cluster signature stability: the author must investigate. The calibrator does not auto-update bugIdentity in gold; `--record-identities` must be explicitly invoked, with a confirmation prompt if existing bugIdentity values differ.

### 5.3 Acceptance thresholds

`acceptance-thresholds.json` (in BugHunter main repo, not bench repo):

## 6. CLI: `bughunter calibrate`

### 6.1 Surface

```
bughunter calibrate
  --app <path>            Path to a bench app directory (contains bughunter.config.json + gold-standard.jsonl)
  --gold <path>           Path to gold-standard.jsonl (default: <app>/gold-standard.jsonl)
  --out <path>            Path to write calibration-report.json (default: <app>/.bughunter/calibration/<isoDate>/)
  --enforce-thresholds    Read acceptance-thresholds.json and exit 1 on per-kind threshold violation
  --thresholds <path>     Override path to acceptance-thresholds.json (default: bundled)
  --record-identities     Run BugHunter, then rewrite gold-standard.jsonl to fill in bugIdentity for structural matches
  --no-boot               Skip bootScript/healthCheck/teardownScript; assume the app is already running
  --json                  Print the calibration-report.json to stdout (in addition to writing to --out)
  --help
```

### 6.2 Exit codes

| Code | Meaning |
|---|---|
| 0 | Calibration completed; thresholds (if enforced) all passed |
| 1 | Calibration completed; one or more thresholds violated (with `--enforce-thresholds`) |
| 2 | Local-environment problem (port collision, missing dependency, app boot failure) |
| 3 | Gold authoring error (invalid JSONL, unknown kind, ambiguous structural match) |
| 4 | BugHunter run failed (partial run, internal error during runRun) |

### 6.3 `--record-identities` semantics

When provided, after the matching step:
- For every gold entry whose match was via structural fallback, take the matched cluster's `bugIdentity` and write it back into the gold entry.
- Pretty-print the rewritten JSONL (one entry per line; key order preserved per Zod schema; trailing newline).
- Print a diff to stdout showing which lines changed.
- If any gold entry already has a `bugIdentity` and it differs from the cluster's, print a warning and require explicit `--force` to overwrite (defensive against silent corpus drift).

## 7. CI / publish workflow

### 7.1 BugHunter main repo: `.github/workflows/calibrate.yml`

```yaml
name: calibrate
on:
  pull_request: { branches: [main] }
  push:         { branches: [main] }
jobs:
  calibrate:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20.11.x' }
      - run: npm ci && npm run build
      - name: Clone BugHunter-bench at pinned SHA
        run: |
          git clone https://github.com/cunninghambe/BugHunter-bench.git /tmp/bench
          cd /tmp/bench && git checkout $(jq -r .pinned /tmp/bench/MANIFEST.json) || true
      - name: Calibrate all 5 apps
        run: |
          for app in vibe-todo vite-shop vue-board next-blog astro-saas; do
            node packages/cli/dist/cli/main.js calibrate \
              --app /tmp/bench/apps/$app \
              --enforce-thresholds \
              --json > /tmp/calibration-$app.json
          done
      - name: Aggregate
        run: node scripts/aggregate-calibration.mjs /tmp/calibration-*.json > /tmp/aggregate.json
      - name: Comment on PR (if PR)
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with: { script: 'require("./scripts/post-calibration-comment.mjs")(github, context, "/tmp/aggregate.json")' }
      - name: Update README (if push to main)
        if: github.event_name == 'push' && github.ref == 'refs/heads/main'
        run: |
          node scripts/update-readme-calibration.mjs /tmp/aggregate.json
          git config user.name 'bughunter-bot' && git config user.email 'bot@bughunter.dev'
          git add README.md && git commit -m 'chore: refresh calibration numbers' || true
          git push origin main
```

## 9. Negative requirements

- Do **not** publish numbers from a partial run.
- Do **not** auto-promote a false-positive cluster to gold.
- Do **not** consolidate the bench repo with internal `fixtures/`.
- Do **not** ship a calibration command that depends on V26 / V27 being merged but does not check at startup that they are actually present (`DETECTOR_REGISTRY` import, `BugCluster.bugIdentity` field) — fail loudly with a clear "requires V26+V27" message if missing.
- Do **not** make per-app thresholds tunable per-PR. The thresholds file is committed; changing it is a deliberate, reviewed action with rationale documented in the file.
- Do **not** add a runtime dep for stats (`simple-statistics`, etc.). Precision/recall/F1 are 4-line formulas.
- Do **not** swallow boot script errors. Any non-zero exit from boot/teardown is fatal exit-code 2.
- Do **not** commit `calibration-report.json` to the BugHunter repo — it is regenerated. Only the README block is committed.
- Do **not** read user-supplied JSONL without Zod validation.
- Do **not** open any external network connection during calibrate. Bench apps are 100% local; if a fixture needs an external API, the bench-repo's `scripts/boot.sh` must spin up an in-process mock.

## 11. Open questions

1. **Should bench apps live in a monorepo (npm workspaces, single `node_modules`) or as 5 standalone repos?** Monorepo is faster to bootstrap (one `npm ci`) but couples versioning. Lean monorepo for V44; revisit if any one app needs a substantially different toolchain version.
2. **Should `bugIdentity` matching use a "near-match" allowance (e.g., a Levenshtein distance on the signature) for cases where a signature drifts slightly between BugHunter versions?** No — exact match only. Drift indicates a real signature-algorithm change and demands `--rebuild-identity` plus a corpus-version-bump. Tolerance hides regressions.
3. **Should the corpus include adversarial bug-free apps (where the right answer is zero clusters)?** Useful for false-positive measurement at scale; defer to a follow-up. V44 measures FP only on the within-bench-app axis, not against a separately-curated bug-free corpus.
4. **Should we publish calibration numbers per-bench-app, or only the aggregate?** README publishes aggregate (less noise, cleaner story). The full per-app `calibration-report.json` is linked from the README and downloadable. Per-app numbers are useful for diagnostics but noisy for marketing.
5. **5 apps enough?** Industry SAST benchmarks use 50–200. 5 is a credible starting point — it covers the four major build-tool / framework combinations (Vite-React, Next, Vue, Astro) plus an outlier (astro-saas with Solid islands). Commit to growing to 20+ over Phase F follow-ups; track in a sibling spec.
6. **How to prevent overfitting to the bench corpus?** The bench corpus is public — anyone can see the deliberate bugs. Mitigations: (a) corpus is **separate** from BugHunter's main repo (commit pressure to tune-to-corpus is partially decoupled), (b) gold authoring requires PR review with `humanRepro` documented, (c) follow-up V45 introduces a *private* held-out corpus that only CI sees.
7. **What threshold should low-confidence kinds (gold count < 3) use?** Currently warning-only. Possible alternatives: (a) report but exclude from aggregate, (b) flag the gap and prioritize gold-authoring. Lean (a) for V44; the aggregate number is the headline.
8. **Should `--no-boot` be the default for CI?** No. CI starts the apps as part of the workflow because the bench repo's boot scripts may evolve; coupling them to the workflow keeps the contract explicit. `--no-boot` exists for local iteration where the developer already has the app running in another terminal.
9. **Should the CI workflow upload `calibration-report.json` as an artifact for download?** Yes — added to §7.1 as `actions/upload-artifact@v4` step. Useful for cross-PR diff and historical comparison even before V45 cross-version regression history lands.
10. **Should bench apps cover the 5 deferred BugKinds?** Yes — every deferred kind gets exactly 1 fixture demonstrating the gap. This is critical for honesty: the deferred-status is *visible* in the report, not implicit.

## 12. Risks + escape hatches

- **Risk: bench app boot is flaky in CI.** Mitigation: every app's `bootScript` polls `healthCheckUrl` with timeout; CI pins Node version; bench-repo CI smoke-tests boot independently every PR. Escape: `--no-boot` allows manual invocation against an already-running app.
- **Risk: gold authoring is slow and PRs to bench repo stall.** Mitigation: `gold-standard.md` companion makes review fast (one paragraph + repro per entry). Escape: PRs that add gold but not fixtures are accepted; the corpus grows organically.
- **Risk: the calibrator becomes the new self-test, displacing V33.** Document the boundary: V33 = internal smoke (one bug per kind, lives in BugHunter repo, runs on every commit). V44 = public benchmark (real-shaped apps, lives in bench repo, runs on PR-to-main and merge). Distinct artifacts.
- **Risk: numbers drop below threshold mid-PR and block legitimate work.** Mitigation: `--enforce-thresholds` is opt-in at the per-PR level (a label `skip-calibration` is honored by the workflow); blocking is the merge-gate, not the PR-iteration gate. Document.
- **Escape hatch:** if the corpus produces persistent noise on one kind that real-world apps never exhibit, bump the per-kind threshold in `acceptance-thresholds.json` with a `rationale` string committed alongside, reviewable on PR.
